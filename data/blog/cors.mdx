CORS(Cross-Origin Resource Sharing)는 추가 HTTP 헤더를 사용하여, 한 출처에서 실행 중인 웹 애플리케이션이 다른 출처의 선택한 자원에 접근할 수 있는 권한을 부여하도록 브라우저에 알리는 체제다. 기본적으로 웹 브라우저는 동일 출처 정책(Same-Origin Policy)을 적용하여, 스크립트가 다른 출처의 자원에 접근하는 것을 제한한다. 이는 웹 애플리케이션의 보안을 강화하기 위한 조치다. 그러나, 때로는 안전한 방법으로 다른 출처의 자원을 요청해야 할 필요가 있으며, 이때 CORS 정책이 사용된다.

### CORS 처리 방법

1. **서버 측 설정**: 서버는 CORS 요청이 수락되었음을 나타내는 `Access-Control-Allow-Origin`과 같은 특정 HTTP 헤더를 응답에 포함시켜야 한다. 이 헤더는 요청을 보낸 출처를 허용하거나, 모든 출처를 허용하는 '*'와 같은 값을 가질 수 있다. 서버가 특정 출처의 요청만을 허용하도록 설정할 수 있으며, 추가적으로 `Access-Control-Allow-Methods`, `Access-Control-Allow-Headers` 등의 헤더를 통해 허용되는 메서드와 헤더를 지정할 수 있다.

2. **프론트엔드에서의 처리**: 프론트엔드 코드 내에서는 특별한 처리가 필요하지 않다. 브라우저가 자동으로 CORS 정책을 적용하며, 서버에서 적절한 CORS 헤더를 반환하지 않는 경우, 브라우저는 자원 접근을 차단한다.

3. **프록시 서버 사용**: CORS 문제를 해결하는 다른 방법으로는 서버 대신 프록시 서버를 경유하여 요청을 전송하는 방법이 있다. 프록시 서버는 동일 출처에서 온 것처럼 요청을 중계함으로써 브라우저의 동일 출처 정책을 우회할 수 있다. 이 방법은 개발 환경에서 자주 사용되지만, 보안 상의 이유로 실제 운영 환경에서는 권장되지 않을 수 있다.

### 요약

CORS는 웹 애플리케이션에서 다른 출처의 자원에 접근할 수 있도록 허용하는 메커니즘이다. 이를 위해 서버는 응답에 적절한 CORS 관련 HTTP 헤더를 포함해야 하며, 클라이언트(브라우저)는 이 헤더를 기반으로 접근을 허용할지 결정한다. 필요에 따라 프록시 서버를 사용하여 CORS 정책을 우회할 수도 있다.
