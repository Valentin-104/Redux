### SSR의 장점

**1. 초기 페이지 로드 성능**

![CSR](/static/images/csr.webp)

- CSR

  - 서버가 빈 HTML 페이지를 클라이언트에 보내고, 브라우저가 HTML코드를 다운로드하여 화면을 렌더링한다.

  - SSR보다 초기 전송되는 페이지의 속도는 빠르지만 서비스에서 필요한 데이터를 클라이언트에서 추가로 요청하여 재구성해야 하기 때문에 전제적인 페이지 완료 시점은 SSR보다 느려진다.

  - 네트워크 상황이 좋지 않다면 CSR을 이용할 경우 사용자는 컨텐츠를 보기 전에 상당 시간 로딩 화면을 봐야 할 수도 있다.

![SSR](/static/images/ssr.webp)

- SSR

  - 반면 SSR은 서버가 요청된 페이지를 처리하여 바로 렌더링할 수 있는 HTML 페이지를 브라우저에 보낸다. 즉, SSR은 서버에서 사용자에게 보여줄 HTML 페이지를 모두 구성하여 사용자에게 페이지를 보여주는 방식이다.

  - SSR을 사용하면 모든 데이터가 매핑된 서비스 페이지를 클라이언트에게 바로 보여줄 수 있다.

  - 서버를 이용해서 완전한 페이지를 구성하기 때문에 클라이언트에서 구성하는 CSR보다 페이지를 구성하는 속도는 늦어지지만, 사진에서 보이는 바와 같이 전체적으로 사용자에게 보여주는 콘텐츠 구성이 완료되는 시점은 빨라진다는 장점이 있다.

**2. 검색 엔진 최적화 (SEO)**

- SSR

  - 검색 엔진이 완전히 렌더링된 페이지 콘텐츠를 크롤링하고 인덱싱할 수 있게 한다. 검색 엔진 크롤러는 종종 JavaScript 기반 콘텐츠에 어려움을 겪기 때문에, 페이지의 검색 가능성과 랭킹을 향상시킬 수 있다.

  - 또한, 데이터를 caching 하지 않고 매 요청마다 새롭게 불러와 HTML을 재생성한다. 그러므로 매 순간 fresh해야하는 데이터들, 예를 들어 뉴스 피드 페이지 같은 데에 SSR을 적용할 수 있다.
